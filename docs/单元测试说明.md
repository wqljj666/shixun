# 单元测试说明

## 本次单元测试任务目的

本次单元测试任务用于验证“线上购药系统”主要业务模块是否按预期工作，覆盖药品信息、购物车与订单、智能药师咨询三个核心模块。测试代码可以帮助发现业务规则遗漏、金额计算错误、状态流转错误和智能咨询风险识别错误。

## 使用的大模型工具

本阶段使用 Codex 辅助分析业务代码、设计测试用例、生成测试类初稿、补充测试方法注释，并根据 `mvn clean test` 的失败信息修正测试代码。

## 使用的测试框架

- JUnit 5
- Mockito
- Spring Boot Test
- MockMvc
- H2 测试数据库配置

## 测试模块选择原因

选择以下模块作为测试重点：

- 药品信息模块：是用户浏览和选购药品的入口，搜索、筛选、详情和处方药标识必须正确。
- 购物车与订单模块：涉及数量、金额、库存、订单状态和支付状态，业务规则较多，适合单元测试验证。
- 智能药师咨询模块：涉及大模型调用、风险识别、多轮会话和免责声明，是课程智能化模块的核心。

## 药品信息模块测试用例说明

测试类：

- `MedicineServiceTest`
- `MedicineControllerTest`

测试内容：

- 根据关键词“维生素”搜索药品，返回匹配药品。
- 根据分类 ID 筛选处方药分类。
- 根据药品 ID 查询详情，返回名称、规格、价格、库存等信息。
- 查询不存在药品时抛出 `BusinessException`。
- 验证 OTC / 处方药字段是否正确。
- 验证药品列表页和详情页 Controller 返回正确视图和模型数据。

## 购物车与订单模块测试用例说明

测试类：

- `CartServiceTest`
- `OrderServiceTest`
- `PaymentServiceTest`
- `OrderControllerTest`

测试内容：

- 普通 OTC 药品可加入购物车。
- 修改购物车数量后，总金额按单价乘数量计算。
- 删除购物车商品时调用删除逻辑。
- 处方药不能直接加入购物车。
- 创建订单后状态为 `WAIT_PAY`。
- 订单金额按药品价格和数量计算。
- 库存不足时不能创建订单。
- 处方药不能直接下单。
- 取消待支付订单成功，并恢复库存。
- 已支付订单不能取消。
- 模拟支付后订单状态变为 `PAID`，并生成支付记录。
- 订单 Controller 能返回确认页、列表页和跳转结果。

## 智能药师咨询模块测试用例说明

测试类：

- `RiskDetectServiceTest`
- `LargeModelServiceTest`
- `AiConsultationServiceTest`
- `AiConsultationControllerTest`

测试内容：

- 普通咨询返回低风险结果。
- 输入“我有青霉素过敏史”可识别为高风险。
- 包含“儿童”“高烧”等关键词时触发高风险。
- 高风险结果中 `needHumanTransfer = true`。
- 多轮连续对话按 `conversationId` 保存用户消息和 AI 回复。
- 可以查询指定会话的历史消息。
- 无 API Key 时，`LargeModelService` 使用本地模拟回复。
- AI 回复包含免责声明：“AI 回答仅供参考，不替代医生诊断或执业药师审核。”
- Controller JSON 接口返回 conversationId、riskLevel 和 needHumanTransfer。

## 大模型在单元测试中的作用

大模型辅助完成：

- 根据业务代码识别测试重点；
- 生成测试类结构；
- 生成 Mockito mock 数据；
- 生成 MockMvc Controller 测试；
- 补充测试方法注释；
- 设计边界条件，例如库存不足、处方药拦截、已支付订单不可取消、高风险咨询转人工。

## 人工如何检查和修正大模型生成的测试代码

人工检查过程包括：

- 运行 `mvn clean test`；
- 查看编译错误和断言失败；
- 修正 MockMvc 测试中与 Spring Boot 3 `jakarta.servlet` 相关的类型问题；
- 修正对实际业务行为的断言，例如智能药师发送一次消息应保存“用户消息 + AI 回复”两条记录；
- 确认测试不会访问真实 DeepSeek API；
- 确认测试中没有真实 API Key。

## 如何运行测试

在项目根目录执行：

```bash
mvn clean test
```
