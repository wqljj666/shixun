# 大模型应用记录

## 在 IDE 或 Codex 中如何使用大模型

开发过程中在 Codex 中输入课程项目需求，由大模型根据需求分析功能模块、生成代码、补充注释、完善页面样式和整理文档。人工负责确认需求、检查运行结果、修正业务规则和确认最终交付内容。

## 大模型生成了哪些代码

大模型辅助生成了以下代码：

- Spring Boot 启动结构；
- 药品信息模块实体、Repository、Service、Controller、VO、DTO；
- 购物车与订单模块实体、Repository、Service、Controller、VO、DTO；
- 智能药师咨询模块实体、Repository、Service、Controller、VO、DTO；
- 订单状态和风险等级枚举；
- Thymeleaf 页面；
- `style.css` 中的统一页面样式；
- H2 初始化测试数据；
- README 和 docs 文档。

## 大模型如何辅助生成方法注释

在最终整理阶段，大模型根据核心 Service 的实际职责生成 JavaDoc 注释。注释内容包含：

- 类的职责；
- 方法功能；
- 入参含义；
- 返回值说明；
- 关键业务规则；
- 异常处理逻辑。

这些注释帮助课程评审者理解业务流程，也体现了“大模型辅助生成方法注释”的使用过程。

## 大模型如何辅助代码补全

大模型根据已有代码上下文自动补全了：

- JPA 实体字段；
- getter / setter；
- Repository 查询方法；
- Service 业务逻辑；
- Controller 路由；
- Thymeleaf 表单和数据显示；
- CSS 组件样式。

## 大模型如何引导生成代码

大模型先根据需求拆分为药品信息、购物车订单和智能药师三个模块，再分别生成实体、数据访问层、业务层、控制器和页面。生成过程中遵循 Spring Boot 常见分层结构，使项目目录清晰、代码职责明确。

智能药师模块中，大模型引导将风险识别、大模型调用和咨询记录保存拆分为不同服务，避免把所有逻辑堆在 Controller 中。

## 人工如何检查和修改大模型生成的代码

人工检查重点包括：

- 业务规则是否满足课程要求；
- 处方药是否被拦截；
- 订单金额是否根据药品价格和数量计算；
- 创建订单是否扣减库存；
- 取消订单是否恢复库存；
- 高风险咨询是否提示转人工；
- AI 回复是否包含免责声明；
- API Key 是否未写死在代码和文档中；
- 页面是否统一、美观；
- `mvn clean test` 是否通过；
- 主要页面是否能访问。

发现问题后，由人工要求大模型继续修改，并通过命令行和页面请求验证修改结果。

## 大模型辅助与传统人工编码方式的比较

与完全人工编码相比，大模型可以更快生成模块骨架、重复性代码和页面样式，尤其适合课程项目中的实体、Repository、Controller、模板页面和文档整理。

但大模型输出仍需要人工检查，特别是：

- 业务规则是否遗漏；
- 安全配置是否正确；
- API Key 是否泄露；
- 高风险医疗咨询是否合规；
- 页面跳转和表单提交是否能真实运行。

因此本项目采用“大模型生成初稿 + 人工检查修正 + 运行验证”的方式完成。

## 对大模型输出内容的验证过程

验证过程包括：

1. 使用 `mvn clean test` 检查项目构建。
2. 使用 `mvn spring-boot:run` 启动项目。
3. 请求首页、药品列表、购物车、订单列表、智能药师、咨询记录和 H2 控制台。
4. 测试药品搜索、分类筛选和详情查看。
5. 测试加入购物车、修改数量、删除商品、创建订单、模拟支付和取消订单。
6. 测试普通智能药师咨询。
7. 测试高风险智能药师咨询。
8. 检查历史咨询记录是否保存。
9. 检查项目中没有真实 API Key。
10. 清理 `target/`、日志和 IDE 配置文件后再提交。
