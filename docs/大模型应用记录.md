# 大模型应用记录

## 在 IDE 或 Codex 中如何使用大模型

开发过程中在 Codex 中输入课程项目需求，由大模型根据需求分析功能模块、生成代码、补充注释、完善页面样式和整理文档。人工负责确认需求、检查运行结果、修正业务规则和确认最终交付内容。

## 大模型生成了哪些代码

大模型辅助生成了以下代码：

- Spring Boot 启动结构；
- 药品信息模块实体、Repository、Service、Controller、VO、DTO；
- 购物车与订单模块实体、Repository、Service、Controller、VO、DTO；
- 智能药师咨询模块实体、Repository、Service、Controller、VO、DTO；
- 智能药师连续多轮对话的会话表、消息表和 AJAX 接口；
- 订单状态和风险等级枚举；
- Thymeleaf 页面；
- `style.css` 中的统一页面样式；
- H2 初始化测试数据；
- README 和 docs 文档。

## 大模型如何辅助生成方法注释

在最终整理阶段，大模型根据核心 Service 的实际职责生成 JavaDoc 注释。注释内容包含：

- 类的职责；
- 方法功能；
- 入参含义；
- 返回值说明；
- 关键业务规则；
- 异常处理逻辑。

这些注释帮助课程评审者理解业务流程，也体现了“大模型辅助生成方法注释”的使用过程。

## 大模型如何辅助代码补全

大模型根据已有代码上下文自动补全了：

- JPA 实体字段；
- getter / setter；
- Repository 查询方法；
- Service 业务逻辑；
- Controller 路由；
- Thymeleaf 表单和数据显示；
- CSS 组件样式。

## 大模型如何引导生成代码

大模型先根据需求拆分为药品信息、购物车订单和智能药师三个模块，再分别生成实体、数据访问层、业务层、控制器和页面。生成过程中遵循 Spring Boot 常见分层结构，使项目目录清晰、代码职责明确。

智能药师模块中，大模型引导将风险识别、大模型调用、咨询会话管理和消息记录保存拆分为不同服务，避免把所有逻辑堆在 Controller 中。后续又将单轮问答重构为 conversationId 驱动的多轮对话结构。

## 人工如何检查和修改大模型生成的代码

人工检查重点包括：

- 业务规则是否满足课程要求；
- 处方药是否被拦截；
- 订单金额是否根据药品价格和数量计算；
- 创建订单是否扣减库存；
- 取消订单是否恢复库存；
- 高风险咨询是否提示转人工；
- 连续追问时是否携带历史上下文；
- 高风险识别是否结合历史上下文；
- AI 回复是否包含免责声明；
- API Key 是否未写死在代码和文档中；
- 页面是否统一、美观；
- `mvn clean test` 是否通过；
- 主要页面是否能访问。

发现问题后，由人工要求大模型继续修改，并通过命令行和页面请求验证修改结果。

## 大模型辅助与传统人工编码方式的比较

与完全人工编码相比，大模型可以更快生成模块骨架、重复性代码和页面样式，尤其适合课程项目中的实体、Repository、Controller、模板页面和文档整理。

但大模型输出仍需要人工检查，特别是：

- 业务规则是否遗漏；
- 安全配置是否正确；
- API Key 是否泄露；
- 高风险医疗咨询是否合规；
- 页面跳转和表单提交是否能真实运行。

因此本项目采用“大模型生成初稿 + 人工检查修正 + 运行验证”的方式完成。

## 对大模型输出内容的验证过程

验证过程包括：

1. 使用 `mvn clean test` 检查项目构建。
2. 使用 `mvn spring-boot:run` 启动项目。
3. 请求首页、药品列表、购物车、订单列表、智能药师、咨询记录和 H2 控制台。
4. 测试药品搜索、分类筛选和详情查看。
5. 测试加入购物车、修改数量、删除商品、创建订单、模拟支付和取消订单。
6. 测试普通智能药师咨询。
7. 测试普通连续咨询。
8. 测试高风险连续咨询。
9. 检查历史咨询会话和多轮消息是否保存。
10. 检查项目中没有真实 API Key。
11. 清理 `target/`、日志和 IDE 配置文件后再提交。

## 单元测试阶段大模型应用记录

### 大模型如何根据业务代码生成测试思路

大模型先阅读药品信息、购物车订单、支付、智能药师咨询等核心 Service 和 Controller，根据业务规则拆分测试点。例如订单模块重点关注订单金额、库存扣减、处方药限制、状态流转；智能药师模块重点关注高风险识别、fallback 回复、多轮会话保存和免责声明。

### 大模型如何生成测试类

大模型按照项目包结构在 `src/test/java/com/example/pharmacy/` 下生成测试类，Service 层优先使用 Mockito，Controller 层使用 MockMvc。测试类包括：

- `MedicineServiceTest`
- `MedicineControllerTest`
- `CartServiceTest`
- `OrderServiceTest`
- `PaymentServiceTest`
- `OrderControllerTest`
- `RiskDetectServiceTest`
- `LargeModelServiceTest`
- `AiConsultationServiceTest`
- `AiConsultationControllerTest`

### 大模型如何生成方法注释

大模型为每个核心测试方法补充了场景说明注释，用于说明该测试验证的业务场景，例如“库存不足时不能创建订单”“高风险咨询应提示转人工药师”“未配置 API Key 时使用本地模拟回复”。

### 大模型如何辅助补充边界条件

大模型辅助补充了以下边界测试：

- 查询不存在药品时抛出业务异常；
- 处方药不能直接加入购物车；
- 处方药不能直接下单；
- 库存不足不能创建订单；
- 已支付订单不能取消；
- 非待支付订单不能重复支付；
- AI 回复必须包含免责声明；
- 高风险识别应忽略 AI 回复中的风险说明，避免误判用户风险。

### 大模型如何辅助分析测试失败原因

第一次运行测试时出现了两类失败：

- MockMvc 自定义视图解析器使用了旧的 mock request/response 类型，不符合 Spring Boot 3 的 `jakarta.servlet` 方法签名；
- 智能药师测试误以为只保存一条消息，但真实业务会保存用户消息和 AI 回复两条记录。

大模型根据 Maven 报错和测试失败栈定位原因，并修正为符合当前业务逻辑的测试断言。

### 人工如何验证测试是否真的有效

人工通过以下方式验证：

- 执行 `mvn clean test`；
- 查看测试总数、失败数和错误数；
- 检查每个测试类是否有明确断言；
- 检查测试是否覆盖主要业务规则；
- 检查测试是否避免真实外部 API 调用；
- 检查测试代码是否未写入真实 API Key。

### 大模型辅助测试与传统人工测试的比较

与传统人工测试相比，大模型能够快速根据代码结构生成测试框架和典型用例，减少重复性 mock 数据编写工作，也能根据报错快速定位测试代码中的不匹配点。

但大模型生成的测试仍需要人工审查，特别是断言是否真的验证业务规则、是否误解了业务行为、是否意外调用外部接口。本项目采用“大模型生成 + 人工检查 + Maven 验证”的方式完成单元测试阶段。
